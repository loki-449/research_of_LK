# CLAUDE.md 编写规范（模板）

> 本文档从 2026-07-13 Session 中实际形成的 CLAUDE.md 三层体系蒸馏而来。
> 用于后续新建子模块或新项目时，快速生成符合 agent 规范的 CLAUDE.md。

---

## 三层体系架构

```
calculation/
├── CLAUDE.md                  ← 第 1 层：项目根（全局规范 + 路由）
├── vasp/
│   └── CLAUDE.md              ← 第 2 层：大型模块入口（子模块组总入口）
├── AIMD/
│   └── CLAUDE.md              ← 第 2 层：独立模块
└── CLAUDE_TEMPLATE.md         ← 本文件（模板参考）
```

**层级规则**：
- 第 1 层：设定全局约定和子模块路由表。不展开具体脚本细节。
- 第 2 层：模块入口 CLAUDE.md，目录下有多个子功能组时按子功能展开。子功能组只需写在模块 CLAUDE.md 里，不单独建文件。
- 模块下若有更细的扩展方向（如 `opt_scf_bader`），在第 2 层 CLAUDE.md 中预留位置即可。

```markdown

- 物理讨论风格（图像优先 → 公式）

# 交互偏好
- 语言
- 注释语言
- 是否自动 Git

# Git / GitHub
- 远程仓库地址
- 分支策略
- 自动推送规则
- 定时任务（如有）
```

## 编写原则（从今天实践中总结）

1. **触发词要具体**：用中文+英文混合列出用户可能说的短语，让 agent 能快速判断"这个模块能处理"。
2. **工作流用实际命令**：不要抽象描述，直接写 `cd <path> && python <script>.py <args>`。
3. **参数要写默认值**：`ENCUT=800` 而不是 "ENCUT 较高"。
4. **跨模块约定要显式声明**：共享库路径、命名规则、工作流顺序依赖，避免 agent 在不同模块间迷失。
5. **物理约定集中在第 1 层**：全局性的计算参数放根 CLAUDE.md，模块特有的放模块 CLAUDE.md。
6. **未来扩展要预留**：在架构图和功能列表中标注"待部署"，让 agent 知道模块还在成长。
7. **注意事项要来自真实踩坑**：如"POTCAR 失败时保留 POSCAR"、"--dt 必须与 POTIM 一致"，不要写空泛的"注意安全"。
